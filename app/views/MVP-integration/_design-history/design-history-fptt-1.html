Design history Integrating with Farm Payments Technical Test

Design history correct as of March 2026

For our private beta we plan to trial the first capabilities in the Farm and Land Service (FLS) (update business and personal details) with users of Farm Payments Technical Test (FPTT), also in private beta.

Conversations between the 2 service teams identified 3 potential integration points:

  • check your details before you apply
  • agreement offer
  • change of circumstance during an agreement

Of the 3, we were able to confirm the first route as an immediate priority.

There’s more work to be done to scope out the other 2 integration points. For agreement offer, to understand if there’s a genuine user need with the expediency at which an offer is made, and for change of circumstance, to scope out the implications of updating details on a user’s compliance with their agreement.

Constraints

Integration at the point of need is the holy grail and what we will aspire to in future. For the initial release, the 2 teams agreed it wouldn’t be possible in the timeframe. We will instead revisit the journeys we designed and tested with another service, where we looked at how users might move between FLS and get funding to improve animal health and welfare (documented in design history v1 and design history v1.1 ).

Pain points and user needs

As with previous integration work, we’re looking to meet the following user need:

“As a farmer or land manager,
I need one place where I can get and manage funding,
So that I do not waste time going to different places.”

Design approach

There are 2 scenarios where we envisage users will need to check and update their details.

Scenario 1 is prior to the application journey. The user receives an email with the actions they need to take before starting an application. Checking and updating details in FLS is one step in the task list.

Scenario 2 is part of the SFI application. The user starts an application and is prompted to check and confirm their details within the FPTT journey. The user is pointed to FLS if any details are incorrect.

There are no plans currently to test the integration journeys. We will look to test again as the cohort of users expands.

What we designed

The 2 scenarios share some pages, with some differences in flows and page content.

Scenario 1

We have captured the scenario 1 journey in Mural (opens in new tab).

Using email as the invitation mechanism, the user is prompted to check their details before starting an FPTT application. Clicking on the link, ‘Update details in the Farm and Land Service’, takes the user to the FLS start page.

A couple of things to note about our start page that have evolved from earlier designs. First, that it only very briefly sets the context of what people can use the service for before the green start button. We’ve kept this intentionally short – we know from previous research that some users want only minimal information and are keen to get on with the task.

Second, is that beneath the green button we have an expanded narrative of FLS’s ambition. We’ve done this to address confusion from some users in the last round of integration testing, with one commenting that they understood FLS to be a rebrand of the Rural Payments service (RPs). The new guidance confirms that FLS will replace RPs, but above and beyond this it will create a single place for any interactions with the government about the user’s farm or land.

From the FLS start page, the user signs in to FLS and chooses their business through the Defra ID screens. They will then come to bp02 SFD business home page, which features a number of enhancements:

  • made headings active and more targeted – ‘Apply for schemes and grants’ to differentiate between applications and active funding and agreements
  • implemented the Tag – GOV.UK Design System (opens in new tab) component to mark the status progress of the application as ‘Not started’
  • added a hyperlink with descriptive text, ‘Start application, within the FPTT signpost
  • improved the link text to bp01 your businesses list – ‘Choose another business’ – to more accurately describe the link location, with some users in earlier integration testing incorrectly thinking this link would return them to the other service

The user then checks and updates their details before navigating back to bp02 SFD business home page via the back link, ‘Back to [business name]’.

We tested 2 iterations of this return journey in our 2 rounds of IAHW user research:

  • Back to Agile Farm Ltd
  • Back to business overview

In the first iteration, all users selected the correct link, with only one showing some hesitation before doing so. There was more hesitation and vocalised confusion when we tested again with the second iteration. As we now have the feasibility of the first link confirmed with our technical team, in these designs we’ve reverted to ‘Back to [business name]’.

Returning to bp02 SFD business home page, the user will then click within the FPTT signpost to ‘Start application, to start the FPTT journey. They do not need to sign in again.

Scenario 2

We have also mapped the scenario 2 journey in Mural (opens in new tab).

The user has already signed in and started their FPTT application. The first screen they encounter asks them to confirm their details to continue their application.

We’ve looked to improve this page by:

  • adding introductory text beneath the H1, ‘If your details are not correct you need to update them before you apply’
  • making naming consistent with FLS
  • reordering information to align with FLS
  • adding a H2 as a call to action, ‘Are these details correct?’
  • using stacked radio buttons to invite the user to actively confirm that their details are correct
  • explicitly stating where the ‘no’ radio button will take the user – to FLS to start the journey of updating details
  • retaining the assisted digital route for those who would prefer to update details offline, with contact details provided in the Details – GOV.UK Design System (opens in new tab) component

Selecting ‘no’ in this journey takes the user directly to FLS bp02 SFD business home page (they do not need to sign in). In this scenario, the page has:

  • headings that are active and targeted – ‘Apply for schemes and grants’
  • the Tag – GOV.UK Design System component, which marks the status progress of the application as ‘In progress’
  • a hyperlink with descriptive text, ‘Continue application, within the FPTT signpost
  • the link text to bp01 your businesses list as ‘Choose another business’

The user selects and updates the details that are incorrect. Once completed, they navigate back to bp02 SFD business home page via ‘Back to [business name]’.

This time on bp02 SFD business home page, as the user has started their application, they select ‘Continue application’. They are then returned to the FPTT page to confirm their details. The updated details will pull through to FPTT, with the user able to select ‘yes’ and resume their application.

Next steps

Delivering the integration is dependent on close alignment between FPTT and SFD teams. As a priority we will identify:

  • all the work needed to deliver the integration
  • any unknowns and dependencies
