Design history Version 14: Doc upload permissions

Why we explored document upload permissions

This post is about the UCD work so far in January 2026 to clarify the different permissions journeys for doc upload, based on the user’s permission level.

In version 11 of the prototype we designed an initial doc upload journey. As usual for our first exploration of a feature, we focused on a journey for a user with permission to do everything (the highest possible level of permission).

We now needed to work out what any journeys might look like for users with lower permission.

This has been challenging because once we started looking closely at the document upload permissions rules provided by RPA, they seemed to conflict with the overall permissions rules we have followed for our permissions designs.

The contradictions we noticed

The doc upload permissions rules suggest that business permission should dictate scheme and land doc upload permissions. We agree that this makes sense for a user who has full business permission, but we do not think it should be the case for users with any other level of business permission. That would conflict with the as-is wider permissions model, and what we have prototyped, which is that if someone has less than full business permission, they can have entirely customised land and scheme permissions. Adding in the doc upload permission rules implies that a user could have seemingly contradictory permissions. For example it would mean:

  • I have ‘make legal changes’ business permission and have no land or scheme permissions. But I have permission to submit documents for land and most schemes.
  • I have ‘view only’ business permission and have ‘submit’ scheme permissions. But I have no permission to view or submit any documents for schemes.

This conflict would be unfeasibly complicated and confusing for both external and internal users, in our strong opinion.

We also queried RPA’s doc upload rule that view=submit - we questioned whether sometimes it may be logical for lower permission users to be able to view documents, even if they should not be able to submit them.

We also noticed that the doc upload categories and queues specified by RPA do not include land, and wondered why. It seems to us land should be included for completeness and consistency.

We have shared all this thinking with RPA colleagues for their feedback.

Journey mapping

While we await clarification, the SFD team decided to proceed for now on the basis of our best assumption of what seems logical, which essentially means continuing with the wider permissions approach where lower permission users have customisable, mix and match permissions for land and schemes.

UCD and BAs thought this out in Mural (opens in new tab). Our assumption is that as a rule, it would seem logical for a:

  • full permission user to be able to submit all documents
  • ‘submit’ scheme permission user to be able to submit and view documents for that scheme
  • ‘amend’ scheme permission user to be able to view and perhaps also submit documents for that scheme (TBC on the submit part)
  • ‘view only’ scheme permission user to only be able to view documents for that scheme

We made similar assumptions about general ‘land’ permissions being reflected in the doc upload land permissions. (Though because ‘land’ is, for unclear reasons, not currently part of RPA’s doc upload categories, this needs some joining the dots).

‘ES’ permissions are also provisionally mapped out as these may vary from other schemes.

Based on these underlying rules, we mapped out design journeys.

Journey A is what we have already prototyped in version 11 - it represents the journey of a user with the highest possible permission levels who can view and submit all documents. This can only be a user with full business permission, because only a user with full business permission can submit documents for business structure changes.

Journey B (opens in new tab) represents the journey of a user with the lowest possible permission levels - the minimum for any user means they cannot view or submit any documents except documents for complaints, probate and appeals. This must be a user with less than full business permission, who also has all low scheme and land permissions.

Journey C (opens in new tab) represents the journey of a user with medium or mixed permission levels. They can view and/or submit more documents than a Journey B user, but not all documents. This must be a user who does not fall into A or B - this means they must have less than full business permission, and they must have anything other than all low scheme and land permissions. For example, a user who can only view/submit for 1 scheme (eg CS) and complaint/probate/appeals. There are more examples of journey C in Mural (opens in new tab) but they need BA checking.

Design notes on journeys B and C

The design approach so far is that:

  • users will only see the documents they are allowed to see listed on the documents page for the business (for example user B will only see complaint, probate or appeal documents, while user C will only see complaint, probate or appeal documents, and CS documents)
  • pages presenting options which are no longer possible can be removed (for example user B does not need to see the ‘Tell us what this relates to’ page as they can no longer choose anything other than 1 option on that page)
  • users will only see options they are allowed to choose (for example user B and user C will not see any option to submit business change evidence, while user C will only see radio buttons for schemes/land they have permission for on ‘Tell us what this relates to’)
  • on check and file view pages, we will still play back the ‘subject’ even if a user has not had to actively select it earlier in the journey (due to them not having any other choice)

Next steps

  • Confirm the rules with RPA, including on view=submit, and whether a user with ‘amend’ for schemes should be able to submit documents
  • Make any changes needed to our proposals
  • Add ‘land’ into the doc upload categories if we can for completeness and check if this requires any amendments to B and C
  • Consider if we want to add a row into the wider permissions selection journey tables to explain what doc upload permissions are given by ‘Submit’, ‘Amend’ and ‘View only’ - though this may not be needed
  • Design, product and BA review of the journeys and examples in Mural and prototype them
  • ALAN and metadata to be done for finalised journeys for initial release
  • User testing should be done on these newest iterations as well as the previous set of iterations from version 14 as all of this is completely untested
  • The design history post for version 11 outlines those previous changes (November 2025) and also at the end contains UR and general recommendations for next steps on doc upload which are still outstanding
