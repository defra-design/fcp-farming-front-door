Design history RPA Land data — what we can and can't do

A pragmatic reference for product, design and engineering conversations about the RPA Land open data feeds. Based on direct probing of the live API in April 2026 against three real SBIs (106332870, 111089693, 113319980).

The RPA Land WFS is a read-only, current-state snapshot of every land parcel, land cover and hedgerow registered against an SBI. There is no version history exposed — historical and superseded rows are filtered out at source. We can show farmers what their land is today, and which parcels were updated since a given date, but we cannot show them what their land used to be or what specifically changed within a parcel.

Changes are atomic at the parcel level: when RPA edits a parcel, every land cover on it gets a new timestamp simultaneously. So "show me which covers were added in 2024" is, in practice, the same query as "show me which parcels were touched in 2024".

1. The dataset

What it is

The RPA (Rural Payments Agency) Land Reference data, served via the RPA API documentation page (opens in new tab) as three OGC Web Feature Service (WFS) endpoints.

Endpoints and feature shape
Dataset Endpoint suffix One row represents
LandParcels /RPA/LandParcels/wfs/ A registered land parcel (MultiPolygon)
LandCovers /RPA/LandCovers/wfs/ A specific cover (Arable Land, Permanent Grassland, Farm Building, etc.) on a parcel
HedgeControl /RPA/HedgeControl/wfs/ A hedgerow segment associated with a parcel (Linestring)

Base URL: https://environment.data.gov.uk/data-services/.

Update cadence

The documentation states the three datasets are updated twice a week, with loads applied within 24 hours of submission.

There is no documented "modified since" endpoint, so any "what's new" feature has to derive that signal from the temporal fields below.

Authentication and access model

  • No API key. Access is gated by SBI (Single Business Identifier).
  • Every request must include cql_filter=SBI=<sbi>; the server short-circuits with "Missing SBI" or "Invalid SBI" before parsing anything else.
  • CORS is open (Access-Control-Allow-Origin: *), so browser-side fetch() works directly. No backend proxy is required for SBI-scoped queries.
  • The /wfs path 308-redirects to /wfs/ (trailing slash) — fine in browsers, a gotcha for curl without -L.

Schema — the fields that matter

All three datasets share these temporal fields with identical names and identical descriptions.

Temporal fields (verbatim from docs)
Field Description
VALID_FROM Date on which the feature became valid
VALID_TO Date on which the feature ceased to be valid
CREATED_ON Date on which the feature was created

Date format: all three are stored as 14-character strings YYYYMMDDhhmmss — for example 20230102094250. They sort lexicographically. The CQL parser accepts ISO literals ('2024-01-01') for comparisons against these fields and does the right thing.

The "still valid" sentinel: VALID_TO = '99991231000000' (year 9999) means "currently valid". It is never NULL on returned rows. To find "active features" you compare against the sentinel, not check for NULL.

Per-dataset identifying fields:

  • LandParcels: SBI, SHEET_ID, PARCEL_ID, AREA_HA, LFA_CODE, ID
  • LandCovers: SBI, SHEET_ID, PARCEL_ID, LAND_COVER_CLASS_CODE, DESCRIPTION, AREA_HA, REFERENCE_PARCELS_ID
  • HedgeControl: SBI, REF_PARCEL_SHEET_ID, REF_PARCEL_PARCEL_ID, LENGTH, ELIGIBLE_FOR_EFA, plus ADJACENT_PARCEL_* fields

2. What you can do (verified working)

Filter by SBI

Mandatory baseline — cql_filter=SBI=<numeric-sbi>.

Filter by PARCEL_ID

Verified working: SBI=… AND PARCEL_ID=6743 returns the cover or parcel rows for that parcel.

Spatial filtering via INTERSECTS

Per the docs: INTERSECTS(geom, POLYGON((…))) is appended to the SBI clause to limit by area-of-interest.

Temporal queries

All three temporal fields are queryable with multiple CQL syntaxes. All return identical row counts (the parser normalises across forms). 

AND CREATED_ON > '2020-01-01'
AND CREATED_ON AFTER 2020-01-01T00:00:00Z
AND CREATED_ON BETWEEN '2020-01-01' AND '2026-04-29'
AND VALID_FROM AFTER 2020-01-01T00:00:00Z

Pagination

count=N&startIndex=M work as documented.

GeoJSON output

outputFormat=application/json returns a standard GeoJSON FeatureCollection with EPSG:4326 (or 27700) coordinates depending on srsname.

3. What you can't do (or have to work around)

No OR in cql_filter

The docs note: "OR cannot be present anywhere in the cql_filter. This will result in a 400 error." Verified — the server rejects compound filters. Anything that would naturally be expressed as a union has to be split into multiple parallel requests and merged client-side.

No historical versions in the dataset

This is the single most important constraint for product framing.

I queried each of the three SBIs for "give me all rows where VALID_TO != '99991231000000'" — i.e. anything superseded.

Historical row counts across three SBIs
SBI Parcels Covers Historical rows
106332870 33 56 0
111089693 104 173 0
113319980 152 861 0

Across 289 physical parcels and 1,090 cover rows, zero non-current rows were returned. This is consistent enough across two orders of magnitude of farm size to be near-certainly a deliberate filter at the publishing layer, not a coincidence of test data.

Practical implications:

  • We cannot show "this cover used to be Arable Land, now it's Permanent Grassland".
  • We cannot show pure deletions — a parcel that was removed simply isn't in the response.
  • We cannot reconstruct a parcel-level timeline of edits.
  • Any "what changed" view is constrained to additions and replacements visible as new currently-valid rows — it can't include "what was removed".

No "modified since" delta endpoint

The docs note that the data refreshes twice a week, but there is no documented way to ask "what changed in the last load". The closest proxy is CREATED_ON > <last-poll-date>, which catches new rows but misses pure removals.

String-quoted CQL values are blocked by the Azure WAF

The endpoint sits behind Microsoft Azure App Gateway, which 403s any request whose URL contains an encoded single quote (%27) around a non-numeric value.

Filter syntax probe results
Filter Result
SHEET_ID=TQ1872 (no quotes) 400 — CQL parser interprets TQ1872 as a property reference
SHEET_ID='TQ1872' (single-quoted, encoded as %27) 403 — WAF blocks
SHEET_ID="TQ1872" (double-quoted) 400 — CQL doesn't accept this form
CREATED_ON > '2020-01-01' (single-quoted date) 200 — works

So string-typed fields can't be filtered server-side at all through this gateway. Date literals slip through because the WAF apparently allows quote+digit patterns but not quote+letter ones.

Workaround: filter on numeric/date fields server-side, then disambiguate by string fields client-side once the (smaller) result set is in hand.

PARCEL_ID is not unique within an SBI

Within a single SBI, two physical parcels on different OS map sheets can share the same numeric PARCEL_ID. Confirmed in SBI 113319980: PARCEL_ID=9420 exists on both SHEET_ID=TQ1872 (3 covers, last touched 2026-03) and SHEET_ID=TQ1470 (1 cover, last touched 2018-11) — two genuinely different parcels.

True identity is (SHEET_ID + PARCEL_ID). Combined with the WAF constraint above, this means production code must filter PARCEL_ID server-side and SHEET_ID client-side — the natural single-query form is unavailable.

Updates are atomic at the parcel level

Within a single (SHEET_ID, PARCEL_ID) physical parcel, every land cover row shares the same VALID_FROM and CREATED_ON — across all 289 parcels we tested, zero exceptions. This means:

  • "Show me which individual covers changed on this parcel" doesn't exist as a meaningful query — they all change together.
  • The "last updated" date is one timestamp per parcel, not per cover.
  • "Date added" per cover is uniform within a parcel — not useful at parcel-detail granularity.
  • "Changes since X" at the SBI level is meaningful (different parcels have different timestamps); at the single-parcel level it's binary (touched or not touched).

No who/why metadata

The schema exposes timestamps but not actor or reason. There is no audit trail — we can't tell a farmer who edited their parcel or what the change was for.

4. How this maps to farmer-facing user stories

User-story matrix
User story Possible? Notes
"Show me my current land — parcels, covers, hedges" Yes The dataset is current-state. Three fetches, GeoJSON, render.
"Show me my land as it was on 1 January 2020" No No historical rows. The current snapshot is the only snapshot.
"Which of my parcels were updated since [date]?" Yes CREATED_ON BETWEEN <date> AND <today>. Each result represents a touched parcel.
"What changed within my parcel TQ1872 9420?" Partial Can answer "yes, it was touched on [date]" but not "X was added, Y was removed".
"When was this parcel last updated?" Yes MAX(VALID_FROM) on the parcel's covers — they're all uniform.
"Was a parcel removed from my farm at some point?" No Removed parcels simply aren't in the response.
"Show me the previous version of this cover before it was changed" No No version history exposed.
"Audit trail: who edited this and why?" No No actor/reason fields.
"Find all my parcels in a specific area" Yes INTERSECTS(geom, POLYGON(…)).
"Real-time updates when RPA changes something" No Twice-weekly refresh, no push, no webhook, no delta endpoint. Polling is the only option.
"Show me what's new on my farm in the last month" Partial Catches additions and edits-as-new-rows; misses pure deletions.
"Total area of arable land on my farm" Yes Sum AREA_HA across covers where DESCRIPTION = 'Arable Land' (filtered client-side, see WAF note).

5. Operational notes

  • Coordinate systems: srsname=EPSG:4326 (WGS84 lat/lon) for browser map use, EPSG:27700 (British National Grid) for analytical work. Both supported.
  • Pagination defaults: server returns up to a default page size; use count= to control, startIndex= to walk.
  • Geometry size: MultiPolygon features for parcels and covers can be large (10–50 KB each for irregular parcels). For SBI 113319980's 861-cover dataset, the response is several MB. Filtering server-side wherever possible matters.
  • Trailing slash on /wfs: 308-redirects to /wfs/. Browsers handle automatically; curl needs -L.
  • Error format: missing/invalid SBI returns a JSON {"error": "…"} with HTTP 400. CQL syntax errors return XML ows:ExceptionReport with HTTP 400.
  • Empirical date span observed across the three test SBIs: 2010 → February 2026. Dates are real and meaningful for "since when" queries on plausibly long horizons.

6. Recommendations for product framing

When briefing stakeholders or designing flows:

  1. Position this as a "current land record" service, not a "land history" service. Anything past-tense beyond "this parcel was last touched on X" is out of reach.
  2. Talk about changes at parcel granularity, not cover granularity. The dataset isn't structured to support cover-level diffs even if the UI suggests otherwise.
  3. Don't promise "what was removed" features. Removed parcels and covers vanish silently from the data. If that's a hard requirement, RPA would need to expose a separate audit feed — that's an upstream conversation, not something we can hack around.
  4. Plan for polling, not push. Twice-weekly refresh, no webhooks. Any "alert me when something changes" feature has to be built on a scheduled job.
  5. Two genuinely useful "change" patterns are accessible today:
    • List-page filter: "Show me only the parcels updated since [date]" — meaningfully narrows large farms (104 parcels → 8 in our tests).
    • Per-parcel last-updated stamp: "This parcel was last edited on [date]" — single, accurate, derived from MAX(VALID_FROM).

Anything richer (timeline, diff, history) needs a different data source.

Appendix — sample CQL queries that are known to work

# Baseline
cql_filter=SBI=106332870

# Time-bound — covers added since 2024
cql_filter=SBI=106332870 AND CREATED_ON BETWEEN '2024-01-01' AND '2026-04-29'

# Specific parcel (server-side numeric, then client-side SHEET_ID filter)
cql_filter=SBI=106332870 AND PARCEL_ID=6743

# Hedge for parcel
cql_filter=SBI=106332870 AND REF_PARCEL_PARCEL_ID=6743

URL-encode the filter (encodeURIComponent in JS). Append &srsname=EPSG:4326&outputFormat=application/json for browser use.
